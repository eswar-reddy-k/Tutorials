📖 Chapter 1

The Theme Park Story

Before diving into code, let's understand the big picture through a fun story.

🎡 Welcome to WonderLand Park!

Imagine you want to visit WonderLand — the most amazing theme park in the world. Here's what happens when you arrive:

🏠 You (👤 User) walk up to the 🏢 Gate / Server and say: "Hi! I'm Alex. I want to enter the park!"

🪪 The guard says: "Sure! Can you prove who you are?" — You show your ID card (your username + password).

🎟️ The guard checks your ID and gives you a Wristband (a Token). This wristband says: "Alex is a VIP guest, valid until 6pm."

🎢 Now, every time you want to ride a rollercoaster, visit a food stall, or enter a VIP zone — you just show your wristband. You don't need to show your ID again!

⏰ At 6pm, the wristband expires. You need to get a new one. This is exactly how JWT (JSON Web Token) works!

🛂 Authentication — "Who are you?"

Authentication is the process of proving your identity. Like showing your ID card at the park entrance. The system asks: "Are you really Alex?"

🔑 Example: Entering your username + password to log in

🎫 Authorization — "What can you do?"

Authorization is about what you're allowed to do after you're identified. Like your VIP wristband letting you into areas basic visitors can't.

🔑 Example: Only admins can delete users

🌐 But wait — why do we need OAuth?

Now imagine you want to share your WonderLand photos to your Instagram account. You'd have to give Instagram your WonderLand password — that's risky! 😰

What if WonderLand had a magic button: "Allow Instagram to post on your behalf"? You click it, WonderLand gives Instagram a special limited key — Instagram can post photos but can't change your password or access your VIP info.

🎉 That magic button is OAuth 2.0 — a way to grant limited access to another application without sharing your password!

🤔 The Problem Before JWT/OAuth

Every API call required sending username + password. Passwords get stolen, sessions got hard to manage across servers. It was messy and dangerous.

💡 The JWT Solution

Issue a signed token after login. The token carries identity info. No need to store sessions on the server. Stateless, fast, portable.

🔗 The OAuth Solution

Delegate access securely. Let apps work together without sharing passwords. Industry-standard, used by Google, Facebook, GitHub, etc.

📦 Chapter 2

The Basics You Need

Don't worry — here's everything explained with super-simple analogies.

📬

HTTP is like sending a letter

HTTP (HyperText Transfer Protocol) is the language browsers and servers use to talk. You send a Request (a letter asking for something), and the server sends back a Response (a letter with the answer).

📤 HTTP Request

You (browser/app) ask the server for something.

GET /api/profile HTTP/1.1
Host: example.com
Authorization: Bearer <token>

🔑 The Authorization header is where JWT tokens travel!

📥 HTTP Response

The server sends back data + a status code.

200 OK
Content-Type: application/json

{
  "name": "Alex",
  "role": "admin"
}

✅ 200=OK · 401=Unauthorized · 403=Forbidden

Key Insight: HTTP is stateless — each request is independent. The server has NO memory of previous requests. This is exactly why we need tokens like JWT!
🍪

Cookies are like a loyalty card

When you log into a website, the server gives your browser a small file called a cookie. Every time you visit, your browser automatically sends this cookie — so the server knows who you are.

🍪 Session-Based Auth (Old Way)

  1. User logs in with username/password
  2. Server creates a session in its database
  3. Server sends a Session ID in a cookie
  4. Browser sends cookie on every request
  5. Server looks up session ID in database

⚠️ Problem: Server must store ALL sessions. Hard to scale!

🪙 Token-Based Auth (Modern Way)

  1. User logs in with username/password
  2. Server creates a JWT and sends it back
  3. Client stores the token (localStorage)
  4. Client sends token in every request header
  5. Server just verifies the token — no DB lookup!

✅ Stateless, scalable, works across domains!

🍽️

An API is like a restaurant menu

An API (Application Programming Interface) is a set of rules that lets two applications talk to each other. The menu lists what you can order. You (the client) order, the kitchen (server) prepares, and the waiter (API) delivers!

Real Example: When a weather app shows your local forecast, it's calling a weather service API: "What's the weather in New York?" The weather service sends data back through the API.

🔒 Why APIs Need Security

Without security, anyone could call your API and access private data, modify records, or delete everything! JWT and OAuth2 are the locks on the API door.

❌ Unsecured API:
GET /api/users/all
Anyone can get all user data!

✅ Secured with JWT:
GET /api/users/all
Authorization: Bearer <JWT>

🏪

Client-Server is like a shop and customer

The Client is you (browser, mobile app) — you make requests. The Server is the shop — it has the data and logic. JWT lets the server hand you a signed "receipt" proving who you are, so you don't re-present your full ID each time.

🏗️ Architecture Overview

📱

The client gets a token from the Auth Server, then uses it to access the Resource Server. The Resource Server verifies the token itself — no extra DB call needed!

🪙 Chapter 3

JWT — JSON Web Tokens

A compact, self-contained way to securely transmit information between parties.

🎟️ Think of JWT as a Magic Wristband

Remember our WonderLand wristband? A JWT is exactly that — but for the internet. It's a small string of text that contains information (like your user ID and role), is digitally signed so no one can fake it, and can be verified instantly without a database call.

🔍 Here's what a real JWT looks like:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyXzEyMyIsIm5hbWUiOiJBbGV4IFNtaXRoIiwicm9sZSI6ImFkbWluIiwiaWF0IjoxNzEzMDAwMDAwLCJleHAiOjE3MTMwMDM2MDB9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
🟠 Orange = Header 🟢 Teal = Payload 🟣 Purple = Signature Separated by dots ( . )
Part 1
📋

Header

Tells the algorithm used to sign the token

{
  "alg": "HS256",
  "typ": "JWT"
}
Part 2
📦

Payload

Contains the actual data (claims)

{
  "sub": "user_123",
  "name": "Alex",
  "role": "admin",
  "exp": 1713003600
}
Part 3
🔏

Signature

Proves the token hasn't been tampered with

HMACSHA256(
  base64(header) + "." +
  base64(payload),
  secretKey
)
🧂 The Signature is Like a Wax Seal: The server creates a unique stamp using a secret key. Anyone can read the payload, but only the server can create a valid seal. If anyone changes even one character in the payload, the seal breaks!

📋 Understanding Claims (Payload Fields)

📌 Registered Claims

Standard fields defined by JWT spec:

  • iss — Issuer (who created it)
  • sub — Subject (who it's about)
  • exp — Expiration time
  • iat — Issued at time

🎯 Public Claims

Collision-resistant names you can define:

  • name — User's name
  • email — Email address
  • role — User's role

🔒 Private Claims

Custom claims agreed upon by both parties:

  • userId — Internal user ID
  • orgId — Organization ID
  • perms — Permission list

🔄 JWT Authentication Flow

1
👤

User Sends Credentials

User sends POST /login with username and password to the server

2
🔍

Server Validates Credentials

Server checks username/password against the database

3
🏭

Server Creates JWT

Server builds header + payload, signs with secret key → JWT token created

4
📤

Token Sent to Client

Server sends the JWT back in the response. Client stores it (localStorage / memory)

5
🚀

Client Makes Authenticated Requests

Every API call includes: Authorization: Bearer <JWT>

6
✅

Server Verifies Token — No DB Needed!

Server re-creates the signature using its secret key and checks if it matches. If yes — access granted!

⚖️ JWT Pros & Cons

  

    

      
✅ Advantages

      
    
        
  • Stateless — no server-side session storage
    • 
        
  • Scalable — works with multiple servers easily
    • 
        
  • Self-contained — carries its own data
    • 
        
  • Cross-domain — works across different domains
    • 
        
  • Fast verification — no database lookup needed
    • 
        
  • Standard format — widely supported
    • 
      

    

    

      
⚠️ Disadvantages

      
    
        
  • Cannot revoke individual tokens before expiry
    • 
        
  • Payload is base64 encoded, NOT encrypted
    • 
        
  • Token size larger than session ID
    • 
        
  • Secret key must be kept safe
    • 
        
  • If stolen, attacker has full access until expiry
    • 
      

    

  


  

    
🛡️ JWT Security Best Practices

    

      
    
        
  • ✅ Always use HTTPS
    • 
        
  • ✅ Set short expiry times (15min–1hr)
    • 
        
  • ✅ Use strong secret keys (256-bit minimum)
    • 
        
  • ✅ Use Refresh Tokens for longer sessions
    • 
      

      
    
        
  • ❌ Never store sensitive data in payload
    • 
        
  • ❌ Avoid localStorage if XSS is a concern
    • 
        
  • ❌ Don't use "none" algorithm
    • 
        
  • ❌ Don't expose secret keys in code repos
    • 
      

    

  







  
🔑 Chapter 4

  
OAuth 2.0 — Delegated Authorization

  
The industry-standard protocol for authorization — powering "Login with Google" and more.


  

    
📸 The Photo Printing Story

    
You want to print your Google Photos using PrintShop (a third-party app). You have two options:

    
❌ Bad way: Give PrintShop your Google username and password. Now they have full access to your email, Drive, everything!

    
✅ OAuth way: Google shows a screen: "PrintShop wants to access your Photos. Allow?" You click Allow. Google gives PrintShop a limited access token — it can only read photos, nothing else. You never share your password!

  


  
👥 The 4 OAuth 2.0 Roles

  

    
👤
Resource Owner
The user who owns the data and grants access
🏠 YOU — the Google user

    
📱
Client
The app that wants to access the resource on your behalf
🖨️ PrintShop app

    
🏦
Authorization Server
Issues tokens after the user grants permission
🔐 Google's OAuth server

    
🗄️
Resource Server
Hosts the protected data — validates tokens
📸 Google Photos API

  


  
🔄 Authorization Code Flow — Step by Step

  
The most secure and commonly used OAuth flow — let's trace "Login with Google":


  

    
1
🖱️
User Clicks "Login with Google"
You click the button on PrintShop's website

    

    
2
📋
Client Sends Authorization Request
PrintShop redirects you to Google's auth page with: client_id, scope=photos, redirect_uri, state

    

    
3
🔐
User Authenticates & Grants Permission
Google shows: "PrintShop wants to access your Google Photos. Allow?" You click Allow.

    

    
4
🎫
Authorization Server Issues Auth Code
Google redirects back to PrintShop's redirect_uri with a short-lived Authorization Code

    

    
5
🔄
Client Exchanges Code for Token
PrintShop's backend sends the code + client_secret to Google's token endpoint. Server-to-server — user never sees this!

    

    
6
🪙
Authorization Server Issues Access Token
Google returns an Access Token (+ optional Refresh Token and ID Token)

    

    
7
📸
Client Accesses Resource with Token
PrintShop calls Google Photos API with the access token and gets your photos!

🪙 OAuth Token Types

  

    
🎫 Authorization Code
A short-lived, one-time code issued after user consent. Immediately exchanged for a real token. Never shown to the browser directly.

    
🪙 Access Token
The actual key used to access APIs. Short-lived (15min–1hr). Sent in every API request header. Often implemented as a JWT!

    
🔄 Refresh Token
Long-lived token used to get new access tokens without re-login. Stored securely server-side. Revocable!

  


  

    
🎯 Understanding Scopes

    
Scopes define what the client application is allowed to do. The user explicitly sees and approves what access is being granted.

    

      

        
Google Scope Examples:

        
    
          
  • 📸 auth/photoslibrary.readonly
    • 
          
  • 📧 auth/gmail.readonly
    • 
          
  • 👤 auth/userinfo.profile
    • 
        

      

      

        
⚡ Principle of Least Privilege

        
Always request only the scopes you actually need! If an app only needs to read photos, don't request Gmail access. This limits damage if the token is compromised.

      

    

  







  
⚖️ Chapter 5

  
JWT vs OAuth 2.0

  
They're different tools that solve different problems — and they often work together!


  

    🧠 Key Insight: JWT is a token format. OAuth 2.0 is an authorization framework. OAuth 2.0 can use JWT as its token format — they're complementary, not competing!
  


  

    
Aspect🪙 JWT🔑 OAuth 2.0

    
      
What is it?A token format / standardAn authorization framework

      
Primary UseCarrying identity & claims securelyDelegating access to third-party apps

      
SolvesStateless authenticationThird-party authorization

      
Token StorageClient-side (localStorage, cookies)Both client and server-side

      
Token Revocation❌ Hard (waits for expiry)✅ Supported (refresh token revocation)

      
ComplexitySimple — just encode/decodeComplex — multiple flows & roles

      
Used for Login?Yes — after verifying credentialsNot alone — needs OpenID Connect

      
Password shared?NeverNever — that's the whole point!

      
Works together?✅ Yes! OAuth2 often uses JWT as its access token format

    
  


  

    

      
🪙 Use JWT When...

      
    
        
  • ✅ You're building your own login system
    • 
        
  • ✅ You need stateless, scalable authentication
    • 
        
  • ✅ You have microservices sharing identity
    • 
        
  • ✅ You want to avoid DB lookups on every request
    • 
        
  • ✅ Your users are your own (no third-party login)
    • 
      

    

    

      
🔑 Use OAuth 2.0 When...

      
    
        
  • ✅ You need "Login with Google/Facebook/GitHub"
    • 
        
  • ✅ A third-party app needs access to your users' data
    • 
        
  • ✅ You're building a platform with an API for others
    • 
        
  • ✅ You need granular, revocable access (scopes)
    • 
        
  • ✅ You're exposing an API to external developers
    •

🧪 Chapter 6

  
Real-World Scenarios

  
Let's see JWT and OAuth2 in action in everyday situations you already use.


  

    

      
📱

      
Logging into a Mobile App

      
A user logs into a food delivery app. The app needs to authenticate the user and keep them logged in securely.

      

1. POST /login {user, pass}

2. Server validates credentials

3. Server returns:

   • access_token (JWT, 15min)

   • refresh_token (7 days)

4. App stores tokens securely

5. Every API call:

   Authorization: Bearer <JWT>

6. Token expires → use refresh

   POST /refresh → new token ✅
      

    

    

      
🌐

      
Accessing APIs Securely

      
A microservices architecture where Service A calls Service B — both must verify identity without a central session store.

      

Service A receives user JWT

↓

Service A calls Service B:

  GET /data

  Authorization: Bearer <JWT>

↓

Service B verifies JWT signature

  (using shared public key)

↓

Service B reads claims from JWT

  sub: user_123, role: admin

↓

Service B returns data ✅

No shared database needed!
      

    

    

      
🔗

      
Login with Google (OAuth + OIDC)

      
A developer tool site lets you "Login with Google" using OAuth 2.0 + OpenID Connect.

      

1. Click "Login with Google"

2. Redirect to Google OAuth

3. Scope: openid profile email

4. User approves

5. Google returns auth code

6. Server exchanges code:

   • access_token (OAuth)

   • id_token (JWT with user info)

7. App reads id_token claims:

   email: alex@gmail.com

   name: Alex Smith

8. User is now logged in! 🎉
      

    

    

      
🤝

      
Third-Party API Integration

      
A calendar app (Calendly) integrates with Google Calendar to create meeting slots automatically.

      

1. "Connect Google Calendar"

2. OAuth scope:

   calendar.events.readwrite

3. User approves on Google

4. Calendly gets:

   • access_token (short-lived)

   • refresh_token (long-lived)

5. Calendly creates events:

   POST calendar.googleapis.com

   Authorization: Bearer token

6. Token expires → refresh

7. User can revoke anytime! 🔒
      

    

  


  

    
🆔 Bonus: OpenID Connect (OIDC)

    
OAuth 2.0 handles authorization, but not authentication. OpenID Connect is a thin identity layer built on top of OAuth 2.0 that adds authentication.

    

      

        
OAuth 2.0 alone

        
✅ "App can access your photos"
❌ "App knows who you are"

      

      

        
OAuth 2.0 + OIDC

        
✅ "App can access your photos"
✅ "App knows who you are"

      

      

        
The ID Token

        
OIDC adds an id_token — a JWT containing the user's identity claims. This powers "Login with Google"!
