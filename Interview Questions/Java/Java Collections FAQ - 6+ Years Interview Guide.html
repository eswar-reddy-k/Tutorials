// ArrayList: DEFAULT choice for most cases
List<String> names = new ArrayList<>();
names.add("Alice");       // O(1)
names.get(0);             // O(1) fast random access
names.add(0, "Bob");      // O(n) shift all elements right

// LinkedList: Only when frequent insert/remove at both ends
Deque<String> deque = new LinkedList<>();
deque.addFirst("First");  // O(1)
deque.addLast("Last");    // O(1)
deque.removeFirst();      // O(1)
In practice, ArrayList wins almost always due to CPU cache locality. Even for insert/remove, array shifting is fast because of contiguous memory. Use LinkedList only as a Deque.

HashMap Internal Working HOTMUST KNOW

How HashMap works step-by-step
HashMap internals: Array of Node (bucket array) Index: [0] [1] [2] [3] [4] [5] [6] [7] ... [15] | | | null Node Node (K1,V1) (K3,V3) | | Node Node (K2,V2) (K4,V4) (collision: (linked list for collisions) linked list) Java 8+: When a bucket has > 8 nodes, linked list -> RED-BLACK TREE (O(n)->O(log n)) 
// Step-by-step: map.put("John", 30)
// 1. Calculate hashCode: "John".hashCode() = 2314539
// 2. Calculate bucket index: hash & (capacity-1) = 2314539 & 15 = 11
// 3. Go to bucket[11]
// 4. If empty: create new Node(hash, "John", 30, null)
// 5. If occupied: traverse linked list
//    - If key found (equals()): replace value
//    - If key not found: add to end (Java 8) or beginning (Java 7)

// Step-by-step: map.get("John")
// 1. Calculate hashCode, then bucket index (same as put)
// 2. Go to bucket[11]
// 3. Compare: first node's hash == key.hashCode() AND key.equals(nodeKey)?
// 4. If match: return value
// 5. If not: traverse to next node. Repeat.

// CRITICAL: This is why equals() and hashCode() contract matters!
// Bad hashCode = all keys in one bucket = O(n) instead of O(1)
PropertyValue
Default capacity16 (must be power of 2)
Load factor0.75 (rehash when 75% full)
Treeify threshold8 (linked list to tree)
Untreeify threshold6 (tree back to list)
GrowthDouble capacity on rehash

Iterator and ListIterator HOT

// Iterator: forward only, all Collections
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.equals("remove_me")) {
        it.remove(); // SAFE removal during iteration!
    }
}

// ListIterator: forward + backward, only Lists
ListIterator<String> lit = list.listIterator();
while (lit.hasNext()) {
    String s = lit.next();
    lit.set("modified");  // Replace current element
    lit.add("new");       // Insert before current position
}
lit.previous(); // Go backward
ConcurrentModificationException: If you modify a collection directly during iteration (using list.remove() instead of iterator.remove()), you get this error! This is called fail-fast behavior.

Generics Basics HOT

What are Generics?
Type parameters that let you write type-safe, reusable code without casting.
// WITHOUT generics (Java 1.4): unsafe, needs casting
List list = new ArrayList();
list.add("Hello");
list.add(42);  // No compile error! Mixed types!
String s = (String) list.get(1); // ClassCastException at RUNTIME!

// WITH generics: type-safe at compile time
List<String> list = new ArrayList<>();
list.add("Hello");
// list.add(42);  // COMPILE ERROR! Only String allowed.
String s = list.get(0); // No cast needed!

// Generic class
public class Box<T> {
    private T item;
    public void set(T item) { this.item = item; }
    public T get() { return item; }
}
Box<String> strBox = new Box<>();
Box<Integer> intBox = new Box<>();

Wildcards: ? extends and ? super HOT

// ? extends T = UPPER bound (read-only, producer)
// "Anything that IS-A T or subclass of T"
void printAll(List<? extends Number> list) {
    for (Number n : list) {
        System.out.println(n); // Can READ as Number
    }
    // list.add(42); // COMPILE ERROR! Can't write (don't know exact type)
}
printAll(List.of(1, 2, 3));     // List<Integer> OK
printAll(List.of(1.0, 2.0));    // List<Double> OK

// ? super T = LOWER bound (write-only, consumer)
// "Anything that IS-A T or superclass of T"
void addNumbers(List<? super Integer> list) {
    list.add(42);    // Can WRITE Integers
    list.add(100);
    // Integer n = list.get(0); // ERROR! Can only get as Object
}
addNumbers(new ArrayList<Number>());  // OK
addNumbers(new ArrayList<Object>());  // OK
PECS: Producer Extends, Consumer Super
If you READ from collection: use ? extends T
If you WRITE to collection: use ? super T
If you do BOTH: use exact type T

Type Erasure

What is it?
Generics are a compile-time feature only. At runtime, type information is ERASED. List<String> becomes List.
// At compile time:
List<String> strings = new ArrayList<>();
List<Integer> ints = new ArrayList<>();

// At runtime (after erasure):
// Both become just List (raw type). Type info is gone!
strings.getClass() == ints.getClass() // TRUE! Both are ArrayList.class

// This is why you CAN'T do:
// new T()          - type unknown at runtime
// new T[]          - can't create generic array
// instanceof List<String>  - type erased

// Queue: FIFO (First In, First Out)
Queue<String> queue = new LinkedList<>();
queue.offer("A"); queue.offer("B"); queue.offer("C");
queue.poll();   // "A" (removes and returns head)

equals() and hashCode() contract for Collections HOT

// If you use custom objects as HashMap keys or in HashSet,
// you MUST override both equals() AND hashCode()

public class Employee {
    private String id;
    private String name;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Employee e)) return false;
        return Objects.equals(id, e.id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id); // MUST use same fields as equals!
    }
}

// What happens without proper equals/hashCode:
Set<Employee> set = new HashSet<>();
set.add(new Employee("1", "John"));
set.add(new Employee("1", "John"));
// Without override: set.size() = 2 (treated as different!)
// With proper override: set.size() = 1 (duplicates detected)

HashMap Rehashing - How it works HOT

// When: size > capacity * loadFactor (default: 16 * 0.75 = 12)
// What happens:
// 1. Create new array of double size (16 -> 32)
// 2. Recalculate bucket index for EVERY entry (hash & (newCapacity-1))
// 3. Move entries to new positions
// This is O(n) and expensive! That's why pre-sizing matters.

HashMap<String, String> map = new HashMap<>(1000); // avoid rehashing!
// Also: loadFactor can be tuned
// Lower (0.5): more memory, fewer collisions, faster get
// Higher (1.0): less memory, more collisions, slower get
Before rehash (capacity=4): After rehash (capacity=8): [0] -> A -> C [0] -> A [1] -> B [1] -> B [2] -> null [2] -> null [3] -> D [3] -> D [4] -> C (rehashed from [0]) [5-7] -> null

How does HashMap handle null key?

// null key ALWAYS goes to bucket[0]
// HashMap has special handling: hash(null) = 0
HashMap<String, String> map = new HashMap<>();
map.put(null, "value");   // Stored at index 0
map.get(null);            // "value"

// Only ONE null key allowed (second put replaces value)
// TreeMap: null key NOT allowed (throws NullPointerException)
//   because it needs to call compareTo() on keys
// ConcurrentHashMap: null key AND null value NOT allowed
//   because null is ambiguous (absent vs null value)

Why HashMap capacity must be power of 2?

// Bucket index calculation: hash & (capacity - 1)
// This is a BITWISE AND, which is much faster than modulo (%)
// But it only works correctly when capacity is power of 2!

// Example: capacity = 16 (binary: 10000), capacity-1 = 15 (binary: 01111)
// hash & 01111 = last 4 bits of hash = index 0-15

// If capacity = 10 (not power of 2):
// capacity-1 = 9 = 01001
// hash & 01001 only uses bits 0 and 3 → uneven bucket distribution!

// HashMap ensures power of 2:
// If you pass capacity=10, it rounds up to 16
// tableSizeFor(10) → 16
