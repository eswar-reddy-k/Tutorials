What is a Class vs Object?

What is it?
A Class is a BLUEPRINT (like an architect's drawing of a house). An Object is the ACTUAL thing built from that blueprint (the real house). 
// Class = Blueprint
public class Car {
    String brand;    // What the car HAS (state/attributes)
    int speed;

    void accelerate() {  // What the car DOES (behavior/methods)
        speed += 10;
    }
}

// Objects = Real things created from blueprint
Car myCar = new Car();       // Object 1
myCar.brand = "Toyota";

Car yourCar = new Car();     // Object 2 (different object, same blueprint)
yourCar.brand = "Honda";
ClassObject
Blueprint/TemplateInstance/Real thing
Defined onceCan create many objects
No memory until instantiatedOccupies memory on heap
Logical entityPhysical entity
Declared with class keywordCreated with new keyword

Constructors in Depth HOT

What is it?
A constructor is a special method that runs AUTOMATICALLY when you create an object with new. Its job is to INITIALIZE the object.

Rules: Same name as class, NO return type (not even void), called automatically. 
public class Employee {
    String name;
    int age;

    // 1. Default Constructor (no-args)
    public Employee() {
        this.name = "Unknown";
        this.age = 0;
    }

    // 2. Parameterized Constructor
    public Employee(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // 3. Copy Constructor (Java doesn't auto-generate this!)
    public Employee(Employee other) {
        this.name = other.name;
        this.age = other.age;
    }
}
Common Mistakes
- If you write ANY constructor, Java does NOT give you the default no-arg constructor anymore.
- Constructor can be private (used in Singleton pattern).
- Constructors are NOT inherited by child classes.
- You CAN'T have both this() and super() in the same constructor.

Constructor Chaining

What is it?
Calling one constructor from another. Use this() for same class, super() for parent class. MUST be the first statement. 
public class Employee {
    String name;
    int age;
    String department;

    public Employee() {
        this("Unknown", 0, "General");  // Calls 3-arg constructor
    }

    public Employee(String name) {
        this(name, 0, "General");       // Calls 3-arg constructor
    }

    public Employee(String name, int age, String dept) {
        this.name = name;               // Final destination - actual initialization
        this.age = age;
        this.department = dept;
    }
}

// Constructor chaining with parent class
class Manager extends Employee {
    int teamSize;
    public Manager(String name, int age, int teamSize) {
        super(name, age, "Management");  // Calls parent constructor FIRST
        this.teamSize = teamSize;
    }
}
Execution order when creating Manager: new Manager("John", 30, 5) | +--> Manager constructor called | +--> super("John", 30, "Management") // Parent constructor runs FIRST | | | +--> Employee fields initialized | +--> this.teamSize = 5 // Then child fields

What is Encapsulation? HOT

What is it?
Encapsulation means wrapping data (variables) and code (methods) together into a single unit (class), and HIDING the internal data from outside access. Only expose controlled access through getters/setters.

Analogy: A medicine CAPSULE. The medicine (data) is inside the capsule (class). You can only take the medicine by swallowing the capsule (using methods), not by opening it and touching the medicine directly. 
// BAD: No encapsulation - data exposed directly
public class BankAccount {
    public double balance;  // Anyone can set balance = -1000000!
}

// GOOD: Proper encapsulation
public class BankAccount {
    private double balance;  // Hidden! Only this class can touch it.

    public double getBalance() {
        return balance;  // Controlled READ access
    }

    public void deposit(double amount) {
        if (amount > 0) {             // VALIDATION!
            balance += amount;
        } else {
            throw new IllegalArgumentException("Amount must be positive");
        }
    }

    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;
        } else {
            throw new IllegalArgumentException("Invalid withdrawal");
        }
    }
}
Why is it used?
1. Data Protection - Prevent invalid states (negative balance)
2. Flexibility - Can change internal implementation without breaking callers
3. Control - Add validation, logging, security in getters/setters
4. Maintenance - Easier to debug because data changes go through one place
Best Practice
- Make ALL fields private
- Provide getters for read access, setters ONLY if modification is needed
- Add validation in setters
- Consider making classes immutable when possible (no setters at all)

What is Inheritance? HOT

What is it?
Inheritance means a child class (subclass) automatically gets all the fields and methods of a parent class (superclass). It's the IS-A relationship.

Analogy: A child inherits features from parents - eye color, height. Similarly, a Dog class inherits from an Animal class. 
// Parent / Superclass / Base class
public class Animal {
    String name;
    int age;

    void eat() {
        System.out.println(name + " is eating");
    }

    void sleep() {
        System.out.println(name + " is sleeping");
    }
}

// Child / Subclass / Derived class
public class Dog extends Animal {
    String breed;

    void bark() {
        System.out.println(name + " says Woof!");
    }
}

// Usage
Dog d = new Dog();
d.name = "Buddy";  // Inherited from Animal
d.eat();             // Inherited from Animal - "Buddy is eating"
d.bark();            // Own method - "Buddy says Woof!"
What is NOT inherited?
- private members (accessible through inherited public/protected methods)
- Constructors (but parent's constructor is called via super())
- Static members belong to the class, not inherited in true sense (they are accessible but not overridden)

Why Java doesn't support Multiple Inheritance with classes? HOT

The Diamond Problem
If two parent classes have the same method, the child wouldn't know WHICH one to use. This ambiguity is called the Diamond Problem.
Animal / \ Flying Swimming \ / Duck <-- Which method does Duck use? Ambiguous! // This would cause ambiguity: class Flying { void move() { "fly"; } } class Swimming { void move() { "swim"; } } class Duck extends Flying, Swimming { } // WHICH move()? COMPILER ERROR!
The Solution: Interfaces
interface Flyable {
    default void move() { System.out.println("Flying"); }
}

interface Swimmable {
    default void move() { System.out.println("Swimming"); }
}

class Duck implements Flyable, Swimmable {
    @Override
    public void move() {
        Flyable.super.move();    // Explicitly choose which
        Swimmable.super.move(); // Or call both
        System.out.println("I do both!");
    }
}

What is Polymorphism? HOT

What is it?
Poly = Many, Morph = Forms. Same method name can behave differently depending on context. Two types:
FeatureCompile-Time (Static)Runtime (Dynamic)
How?Method OverloadingMethod Overriding
BindingEarly binding (at compile time)Late binding (at runtime)
Which method?Decided by parameter typesDecided by actual object type
Same class?Yes (same class, diff params)No (parent-child relationship)
Exampleadd(int,int) vs add(double,double)animal.sound() = "Bark" or "Meow"
// RUNTIME POLYMORPHISM (most asked)
class Animal {
    void sound() { System.out.println("Some sound"); }
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Bark!"); }
}

class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow!"); }
}

// The MAGIC: Parent reference, child object
Animal a1 = new Dog();  // Reference type: Animal, Actual object: Dog
Animal a2 = new Cat();

a1.sound();  // "Bark!" - JVM sees the ACTUAL object (Dog) at runtime
a2.sound();  // "Meow!" - JVM sees the ACTUAL object (Cat) at runtime

// Real-world usage: Process different types uniformly
List<Animal> animals = List.of(new Dog(), new Cat(), new Dog());
for (Animal a : animals) {
    a.sound();  // Each calls its OWN implementation!
}

Method Overloading (Compile-Time Polymorphism) HOT

What is it?
Multiple methods with the SAME NAME but DIFFERENT parameters in the same class. 
public class Calculator {
    // Overloaded methods - same name, different params
    int add(int a, int b)                    { return a + b; }
    double add(double a, double b)            { return a + b; }
    int add(int a, int b, int c)              { return a + b + c; }
    String add(String a, String b)             { return a + b; }
}

// Compiler decides which to call based on arguments:
calc.add(1, 2);         // Calls int version
calc.add(1.5, 2.5);     // Calls double version
calc.add(1, 2, 3);      // Calls 3-arg version
Rules and Tricky Cases
You CAN overload by: Number of params, Type of params, Order of params
You CANNOT overload by: Return type alone, Access modifier alone

Tricky: add(int, long) and add(long, int) ARE valid overloads. But calling add(1, 1) causes ambiguity error!

Method Overriding (Runtime Polymorphism) HOT

What is it?
Child class provides its OWN implementation of a method that already exists in the parent class. Same name, same parameters, same return type (or covariant). 
class Vehicle {
    void start() {
        System.out.println("Vehicle starting...");
    }
}

class ElectricCar extends Vehicle {
    @Override  // ALWAYS use @Override annotation!
    void start() {
        System.out.println("Electric car starting silently...");
    }
}
Rules of Overriding:
RuleExplanation
Same method signatureName + parameters must match exactly
Return typeSame or covariant (subtype) return type
Access modifierSame or WIDER (protected in parent -> can be public in child)
ExceptionsCan throw same, narrower, or no exception (NOT broader)
static methodsCannot be overridden (only hidden)
final methodsCannot be overridden
private methodsCannot be overridden (not visible to child)
ConstructorsCannot be overridden

Covariant Return Type

// Parent returns Animal, Child can return Dog (subtype of Animal)
class AnimalFactory {
    Animal create() { return new Animal(); }
}

class DogFactory extends AnimalFactory {
    @Override
    Dog create() { return new Dog(); }  // Dog IS-A Animal, so this is valid!
}

// This avoids casting at the call site:
DogFactory factory = new DogFactory();
Dog dog = factory.create();  // No cast needed!

What is Abstraction? HOT

What is it?
Abstraction means showing only WHAT something does, hiding HOW it does it. You define the contract (what methods exist) without worrying about implementation details.

Analogy: When you drive a car, you use the steering wheel, pedals, and gear shift. You don't need to know how the engine combustion works internally. The steering wheel is the ABSTRACTION.
Two ways to achieve abstraction in Java:
Abstract ClassInterface
0 to 100% abstraction100% abstraction (before Java 8)
Can have concrete methodsCan have default/static methods (Java 8+)
Can have state (fields)Only constants (public static final)
extends (single)implements (multiple)

Abstract Class in Depth HOT

public abstract class Shape {
    String color;  // CAN have instance fields

    public Shape(String color) {  // CAN have constructors
        this.color = color;
    }

    abstract double area();      // Abstract: NO body, child MUST implement
    abstract double perimeter(); // Abstract: NO body

    void display() {             // Concrete: HAS body, child inherits
        System.out.println("Color: " + color + ", Area: " + area());
    }
}

public class Circle extends Shape {
    double radius;

    public Circle(String color, double radius) {
        super(color);  // Call abstract class constructor
        this.radius = radius;
    }

    @Override
    double area() { return Math.PI * radius * radius; }

    @Override
    double perimeter() { return 2 * Math.PI * radius; }
}

// Cannot instantiate abstract class:
// Shape s = new Shape("Red"); // COMPILE ERROR!
Shape s = new Circle("Red", 5.0);  // OK: Polymorphism!
s.display();  // "Color: Red, Area: 78.54"

Interfaces in Depth HOT

// Interface = 100% contract/specification
public interface PaymentGateway {

    // Constants (implicitly public static final)
    String VERSION = "2.0";

    // Abstract method (implicitly public abstract)
    boolean processPayment(double amount);
    void refund(String transactionId);

    // Default method (Java 8+) - has body, can be overridden
    default void log(String msg) {
        System.out.println("[Payment] " + msg);
    }

    // Static method (Java 8+) - belongs to interface, can't be overridden
    static PaymentGateway create(String type) {
        return switch (type) {
            case "stripe"  -> new StripePayment();
            case "paypal"  -> new PayPalPayment();
            default        -> throw new IllegalArgumentException("Unknown");
        };
    }

    // Private method (Java 9+) - helper for default methods
    private void validate(double amount) {
        if (amount <= 0) throw new IllegalArgumentException();
    }
}

// Implementing
public class StripePayment implements PaymentGateway {
    @Override
    public boolean processPayment(double amount) {
        // Stripe-specific logic
        return true;
    }

    @Override
    public void refund(String txnId) {
        // Stripe-specific refund
    }
}

Abstract Class vs Interface HOTMUST KNOW

FeatureAbstract ClassInterface
Keywordabstract classinterface

When to use which?
Use Abstract Class when:
- You want to share CODE among related classes (template method pattern)
- You need non-static/non-final fields
- Classes share IS-A relationship (Dog IS-A Animal)

Use Interface when:
- Unrelated classes need same capability (Comparable, Serializable)
- You need multiple inheritance
- You want to define a CONTRACT without caring about implementation
- Strategy pattern, dependency injection

Default Methods in Interfaces (Java 8+)

Why were they added?
To add new methods to interfaces WITHOUT breaking existing implementations. Before Java 8, adding a method to an interface would break ALL classes implementing it. 
public interface Collection<E> {
    // This was added in Java 8 without breaking ArrayList, LinkedList, etc.
    default Stream<E> stream() {
        return StreamSupport.stream(spliterator(), false);
    }

    default void forEach(Consumer<? super E> action) {
        for (E e : this) { action.accept(e); }
    }
}

Functional Interfaces HOT

What is it?
An interface with exactly ONE abstract method. Used as the basis for lambda expressions. 
@FunctionalInterface
public interface Comparator<T> {
    int compare(T o1, T o2);  // Only ONE abstract method
    // Can have default and static methods too
}

// Key built-in functional interfaces:

Predicate<T>test(T)TbooleanFiltering
Function<T,R>apply(T)TRTransformation
Consumer<T>accept(T)TvoidProcessing (forEach)
Supplier<T>get()noneTFactory/Lazy init
Runnablerun()nonevoidThread execution
Callable<V>call()noneVThread with return
// Lambda = shorter way to implement functional interface
Predicate<String> isLong = s -> s.length() > 5;
Function<String, Integer> toLength = String::length;
Consumer<String> printer = System.out::println;
Supplier<List<String>> listFactory = ArrayList::new;

Marker Interfaces

What is it?
An interface with NO methods. It "marks" a class as having a certain property. JVM/frameworks check instanceof to see if a class is marked. 
// Built-in Marker Interfaces
public interface Serializable { }    // Marks: "can be serialized"
public interface Cloneable { }       // Marks: "can be cloned"
public interface RandomAccess { }    // Marks: "supports fast random access"

// Custom marker interface
public interface Auditable { }

public class Order implements Auditable {
    // Now we can check: if (obj instanceof Auditable) { log changes }
}

// Modern alternative: Use ANNOTATIONS instead
@Retention(RetentionPolicy.RUNTIME)
public @interface Auditable { }  // Cleaner approach!

What is this keyword?

public class Employee {
    String name;

    // 1. Refer to current instance variable (most common)
    public Employee(String name) {
        this.name = name;  // this.name = field, name = parameter
    }

    // 2. Call another constructor (constructor chaining)
    public Employee() {
        this("Unknown");  // Calls the parameterized constructor
    }

    // 3. Pass current object as argument
    void register() {
        Registry.add(this);  // Pass myself
    }

    // 4. Return current object (method chaining / fluent API)
    Employee setName(String name) {
        this.name = name;
        return this;  // Enables: emp.setName("A").setAge(25)
    }
}

What is super keyword?

class Animal {
    String name;
    Animal(String name) { this.name = name; }
    void eat() { System.out.println("Animal eating"); }
}

class Dog extends Animal {
    String breed;

    // 1. Call parent constructor
    Dog(String name, String breed) {
        super(name);  // MUST be first line. Calls Animal(name)
        this.breed = breed;
    }

    // 2. Call parent method
    @Override
    void eat() {
        super.eat();  // Call parent's eat() first
        System.out.println("Dog eating kibble");
    }

    // 3. Access parent field (if shadowed)
    void show() {
        System.out.println(super.name);  // Parent's name field
    }
}

Composition vs Inheritance HOTMUST KNOW

What is Composition?
Inheritance = IS-A relationship (Dog IS-A Animal)
Composition = HAS-A relationship (Car HAS-A Engine)

The famous advice: "Favor Composition over Inheritance" 
// INHERITANCE approach (tight coupling - fragile!)
class Car extends Engine { }  // Car IS-A Engine? Doesn't make sense!

// COMPOSITION approach (flexible - preferred!)
class Car {
    private Engine engine;       // Car HAS-A Engine. Makes sense!
    private Transmission trans;   // Car HAS-A Transmission

    public Car(Engine engine, Transmission trans) {
        this.engine = engine;      // Injected - easy to swap!
        this.trans = trans;
    }

    void start() {
        engine.ignite();
        trans.engage();
    }
}
FeatureInheritanceComposition
RelationshipIS-AHAS-A
CouplingTight (child depends on parent)Loose (can swap components)
FlexibilityFixed at compile timeCan change at runtime
EncapsulationBreaks (child sees parent internals)Preserves (uses only public API)
TestingHarder to mock parentEasy to mock components
When to useTrue IS-A + want to reuse codeAlmost always (default choice)

Association, Aggregation, and Composition

Association (uses-a): Teacher ----> Student (loose, independent) Aggregation (has-a): Team o---> Player (whole-part, player can exist alone) Composition (owns-a): House *---> Room (strong, room dies with house)
TypeRelationshipLifecycleExample
AssociationUses-AIndependentTeacher uses Student
AggregationHas-A (weak)Child can outlive parentTeam has Players (players exist without team)
CompositionHas-A (strong)Child dies with parentHouse has Rooms (rooms don't exist without house)
// Aggregation: Department HAS Employees (employees can exist without dept)
class Department {
    List<Employee> employees;  // Employees passed in, NOT created here
    Department(List<Employee> employees) {
        this.employees = employees;
    }
}

// Composition: Car OWNS Engine (engine created and destroyed with car)
class Car {
    private final Engine engine = new Engine();  // Created HERE, dies with Car
}

Coupling vs Cohesion

ConceptWhat it meansGoal
CouplingHow much one class DEPENDS on anotherLOW coupling (less dependency)
CohesionHow focused a class is on ONE responsibilityHIGH cohesion (focused)
// HIGH coupling (BAD) - OrderService directly creates EmailService
class OrderService {
    EmailService email = new EmailService();  // Tightly coupled!

class OrderService {
    private final NotificationService notifier;  // Interface!

    void createUser() {}
    void sendEmail() {}
    void generateReport() {}
    void processPayment() {}  // Too many responsibilities!
}

// HIGH cohesion (GOOD) - Each class does ONE thing
class UserService { void createUser() {} }
class EmailService { void sendEmail() {} }
class ReportService { void generateReport() {} }

SOLID - Single Responsibility Principle (SRP) HOT

What is it?
A class should have only ONE reason to change. One class = One job. 
// BAD: Employee class does too much
class Employee {
    void calculatePay() { }      // Payroll logic
    void saveToDatabase() { }    // Persistence logic
    void generateReport() { }    // Reporting logic
}

// GOOD: Split responsibilities
class Employee { /* only employee data */ }
class PayrollService { void calculatePay(Employee e) { } }
class EmployeeRepository { void save(Employee e) { } }
class EmployeeReportGenerator { void generate(Employee e) { } }

SOLID - Open/Closed Principle (OCP) HOT

What is it?
Open for EXTENSION, Closed for MODIFICATION. Add new features by adding new code, not changing existing code. 
// BAD: Must modify this method every time a new shape is added
double calculateArea(Object shape) {
    if (shape instanceof Circle) return ...;
    else if (shape instanceof Rectangle) return ...;
    // Must add new if-else for every new shape!
}

// GOOD: Use abstraction. New shapes = new class, no modification needed
interface Shape {
    double area();
}
class Circle implements Shape { public double area() { ... } }
class Rectangle implements Shape { public double area() { ... } }
class Triangle implements Shape { public double area() { ... } }
// Added Triangle WITHOUT modifying existing code!

SOLID - Liskov Substitution Principle (LSP)

What is it?
A child class should be able to replace its parent class WITHOUT breaking the program. 
// BAD: Violates LSP - Square changes Rectangle's behavior
class Rectangle {
    int width, height;
    void setWidth(int w) { width = w; }
    void setHeight(int h) { height = h; }
    int area() { return width * height; }
}

class Square extends Rectangle {
    @Override
    void setWidth(int w) { width = w; height = w; }  // Surprise behavior!
    @Override
    void setHeight(int h) { width = h; height = h; }
}

// This breaks when substituting:
Rectangle r = new Square();
r.setWidth(5);
r.setHeight(3);
r.area();  // Expected 15 (5x3), got 9 (3x3)! BROKEN!

// GOOD: Use separate classes or a common Shape interface

SOLID - Interface Segregation Principle (ISP)

What is it?
Don't force classes to implement interfaces they don't use. Split large interfaces into smaller, focused ones. 
// BAD: Fat interface - not all workers eat at the factory
interface Worker {
    void work();
    void eat();
    void sleep();
}

class Robot implements Worker {
    void work() { /* OK */ }
    void eat() { /* Robots don't eat! Forced to implement! */ }
    void sleep() { /* Robots don't sleep! */ }
}

// GOOD: Split into focused interfaces
interface Workable { void work(); }
interface Eatable  { void eat(); }
interface Sleepable { void sleep(); }

class Human implements Workable, Eatable, Sleepable { ... }
class Robot implements Workable { ... }  // Only what it needs!

SOLID - Dependency Inversion Principle (DIP) HOT

What is it?
High-level modules should NOT depend on low-level modules. Both should depend on ABSTRACTIONS (interfaces). 
// BAD: High-level depends on low-level directly
class OrderService {
    private MySQLDatabase db = new MySQLDatabase();  // Locked to MySQL!
    void save(Order order) { db.insert(order); }
}

// GOOD: Both depend on abstraction
interface OrderRepository {
    void save(Order order);
}

class MySQLOrderRepo implements OrderRepository { ... }
class MongoOrderRepo implements OrderRepository { ... }

class OrderService {
    private final OrderRepository repo;  // Depends on INTERFACE!

    OrderService(OrderRepository repo) {
        this.repo = repo;  // Inject any implementation!
    }
}

// Spring does this automatically with @Autowired!

Singleton Design Pattern HOT

What is it?
Ensures a class has ONLY ONE instance. Used for shared resources like DB connections, configuration, caches. 
// Thread-safe Singleton using Bill Pugh approach (BEST)
public class DatabaseConnection {
    private DatabaseConnection() { }  // Private constructor!

    private static class Holder {
        private static final DatabaseConnection INSTANCE =
            new DatabaseConnection();
    }

    public static DatabaseConnection getInstance() {
        return Holder.INSTANCE;  // Lazy + Thread-safe (class loading guarantee)
    }
}

// Even simpler: Enum Singleton (Joshua Bloch recommended)
public enum DatabaseConnection {
    INSTANCE;

    public void connect() { /* ... */ }
}
// Usage: DatabaseConnection.INSTANCE.connect();
// Handles serialization, reflection, and thread safety for FREE!
Ways to break Singleton and prevention
1. Reflection - Can call private constructor. Prevent: throw exception in constructor if instance exists.
2. Serialization - Creates new instance on deserialization. Prevent: implement readResolve().
3. Cloning - Can clone the instance. Prevent: throw exception in clone().
4. Enum Singleton prevents ALL of these automatically!

Factory Design Pattern HOT

// Interface
interface Notification {
    void send(String message);
}

// Implementations
class EmailNotification implements Notification {
    public void send(String msg) { System.out.println("Email: " + msg); }
}
class SMSNotification implements Notification {
    public void send(String msg) { System.out.println("SMS: " + msg); }
}
class PushNotification implements Notification {
    public void send(String msg) { System.out.println("Push: " + msg); }
}

// Factory - hides creation logic
class NotificationFactory {
    public static Notification create(String type) {
        return switch (type.toLowerCase()) {
            case "email" -> new EmailNotification();
            case "sms"   -> new SMSNotification();
            case "push"  -> new PushNotification();
            default      -> throw new IllegalArgumentException("Unknown");
        };
    }
}

// Usage: Client doesn't know or care about concrete classes
Notification n = NotificationFactory.create("email");
n.send("Hello!");

Builder Design Pattern HOT

Why?
When a class has many optional parameters, constructors become a mess (telescoping constructor anti-pattern). Builder solves this. 
public class HttpRequest {
    private final String url;
    private final String method;
    private final Map<String,String> headers;
    private final String body;
    private final int timeout;

    private HttpRequest(Builder b) {
        this.url = b.url; this.method = b.method;
        this.headers = b.headers; this.body = b.body;
        this.timeout = b.timeout;
    }

    public static class Builder {
        private final String url;      // Required
        private String method = "GET"; // Default
        private Map<String,String> headers = new HashMap<>();
        private String body;
        private int timeout = 30000;

        public Builder(String url) { this.url = url; }

        public Builder method(String m) { method = m; return this; }
        public Builder header(String k, String v) { headers.put(k,v); return this; }
        public Builder body(String b) { body = b; return this; }
        public Builder timeout(int t) { timeout = t; return this; }

        public HttpRequest build() { return new HttpRequest(this); }
    }
}

// Clean, readable usage:
HttpRequest req = new HttpRequest.Builder("https://api.com/users")
    .method("POST")
    .header("Content-Type", "application/json")
    .body("{\"name\":\"John\"}")
    .timeout(5000)
    .build();

Observer Design Pattern

What is it?
When one object changes, all its dependents are notified automatically. Like subscribing to a YouTube channel - you get notified of every new video. 
// Observer interface
interface OrderListener {
    void onOrderPlaced(Order order);
}

// Subject (Publisher)
class OrderService {
    private List<OrderListener> listeners = new ArrayList<>();

    void subscribe(OrderListener l) { listeners.add(l); }

    void placeOrder(Order order) {
        // ... save order ...
        listeners.forEach(l -> l.onOrderPlaced(order)); // Notify all!
    }
}

// Observers (Subscribers)
class EmailNotifier implements OrderListener {
    public void onOrderPlaced(Order o) { /* send email */ }
}
class InventoryUpdater implements OrderListener {
    public void onOrderPlaced(Order o) { /* update stock */ }
}

// Wire up
OrderService service = new OrderService();
service.subscribe(new EmailNotifier());
service.subscribe(new InventoryUpdater());
service.placeOrder(order); // Both get notified!

Strategy Design Pattern HOT

What is it?
Define a family of algorithms, put each in a separate class, and make them interchangeable at runtime. 
// Strategy interface
interface SortStrategy {
    void sort(int[] array);
}

// Concrete strategies
class BubbleSort implements SortStrategy {
    public void sort(int[] arr) { /* bubble sort logic */ }
}
class QuickSort implements SortStrategy {
    public void sort(int[] arr) { /* quick sort logic */ }
}

// Context class
class Sorter {
    private SortStrategy strategy;
    void setStrategy(SortStrategy s) { strategy = s; }
    void sort(int[] arr) { strategy.sort(arr); }
}

// Usage - swap algorithms at runtime!
Sorter sorter = new Sorter();
sorter.setStrategy(new BubbleSort());  // Small data
sorter.sort(smallArray);

sorter.setStrategy(new QuickSort());   // Large data
sorter.sort(largeArray);

How to create an Immutable Class? HOT

public final class Money {                       // 1. final class (can't extend)
    private final BigDecimal amount;                // 2. final fields
    private final String currency;
    private final List<String> tags;

    public Money(BigDecimal amount, String currency, List<String> tags) {
        this.amount = amount;
        this.currency = currency;
        this.tags = List.copyOf(tags);               // 3. Defensive copy of mutables!
    }

    public BigDecimal getAmount() { return amount; }  // 4. Only getters, NO setters
    public String getCurrency() { return currency; }
    public List<String> getTags() {
        return Collections.unmodifiableList(tags);   // 5. Return unmodifiable copy!
    }

    public Money add(Money other) {
        return new Money(amount.add(other.amount), currency, tags);  // 6. Return NEW object
    }
}
5 Rules for Immutability:
#RuleWhy
1Make class finalPrevent subclass from breaking immutability
2Make all fields private finalCan't access or reassign
3No settersNo way to modify state
4Defensive copy in constructorCaller can't modify via original reference
5Return copies from gettersCaller can't modify via returned reference

Diamond Problem in Java 8+ Interfaces

interface A {
    default void greet() { System.out.println("Hello from A"); }
}
interface B extends A {
    default void greet() { System.out.println("Hello from B"); }
}
interface C extends A {
    default void greet() { System.out.println("Hello from C"); }
}

// Diamond! Must resolve explicitly.
class D implements B, C {
    @Override
    public void greet() {
        B.super.greet();  // Choose B's version
    }
}
Resolution Rules: Class method wins over interface default > Most specific interface wins > Must override if ambiguous.

Upcasting vs Downcasting

// UPCASTING: Child -> Parent (automatic, safe)
Dog dog = new Dog();
Animal animal = dog;  // Dog IS-A Animal. Always safe.

// DOWNCASTING: Parent -> Child (manual, risky)
Animal a = new Dog();
Dog d = (Dog) a;       // OK - actual object IS a Dog

Animal a2 = new Cat();
Dog d2 = (Dog) a2;     // RUNTIME ERROR! ClassCastException! Cat is NOT a Dog!

// SAFE downcasting with instanceof
if (a2 instanceof Dog dog2) {
    dog2.bark();  // Safe!
}

Can a constructor be private?

YES! Used in:
- Singleton pattern (only one instance)
- Factory pattern (objects created via static methods)
- Utility classes (like Math, Collections - all static methods, no instances needed)

Method TypeCan Override?Why?

Can an abstract class have a constructor?

YES! Even though you can't instantiate an abstract class directly, the constructor is called when a concrete subclass is created (via super()). It's used to initialize shared state. 
abstract class Shape {
    String color;
    Shape(String color) { this.color = color; }  // Called by subclass
}

class Circle extends Shape {
    Circle() { super("Red"); }  // Calls Shape's constructor
}
44

Can an interface have a constructor?

NO! Interfaces cannot have constructors because:
- They cannot have instance state (fields are static final)
- They cannot be instantiated
- A class can implement multiple interfaces (which constructor would run first?)
