try-catch-finally HOT

try {
    // Code that might throw exception
    int result = 10 / 0;
} catch (ArithmeticException e) {
    // Handle specific exception
    System.out.println("Can't divide by zero: " + e.getMessage());
} catch (Exception e) {
    // Handle broader exception (must be AFTER specific ones!)
    System.out.println("Error: " + e.getMessage());
} finally {
    // ALWAYS executes (cleanup code)
    // Even if try succeeds, even if catch has return statement
    System.out.println("Cleanup done");
}
When does finally NOT execute?
1. System.exit(0) called in try/catch
2. JVM crashes
3. Infinite loop in try/catch
4. Thread is killed
That's it! Every other case, finally ALWAYS runs.

try-with-resources (Java 7+) HOTMUST KNOW

What is it?
Automatically closes resources that implement AutoCloseable/Closeable. No need for finally block!
// OLD way: verbose, error-prone
FileReader fr = null;
try {
    fr = new FileReader("file.txt");
    // use fr
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (fr != null) {
        try { fr.close(); } catch (IOException e) { /* swallowed! */ }
    }
}

// NEW way: clean, safe, auto-close
try (FileReader fr = new FileReader("file.txt");
     BufferedReader br = new BufferedReader(fr)) {
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
}
// fr and br are AUTOMATICALLY closed! Even if exception occurs.
// Closed in REVERSE order of creation (br first, then fr)

// Java 9+: Can use effectively-final variables
FileReader fr = new FileReader("file.txt");
try (fr) { // fr is effectively final
    // use fr
}
ALWAYS use try-with-resources for: streams, connections, readers, writers, sockets, channels.

Custom Exceptions HOT

// Custom CHECKED exception (extends Exception)
public class InsufficientFundsException extends Exception {
    private final double amount;

    public InsufficientFundsException(String message, double amount) {
        super(message);
        this.amount = amount;
    }

    public double getAmount() { return amount; }
}

// Custom UNCHECKED exception (extends RuntimeException)
public class UserNotFoundException extends RuntimeException {
    private final String userId;

    public UserNotFoundException(String userId) {
        super("User not found: " + userId);
        this.userId = userId;
    }

    public String getUserId() { return userId; }
}

// Usage
void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException(
            "Insufficient funds for withdrawal of $" + amount, amount);
    }
    balance -= amount;
}

User findUser(String id) {
    return userRepo.findById(id)
        .orElseThrow(() -> new UserNotFoundException(id));
}
When to use Checked vs Unchecked custom exceptions?
- Checked: Caller CAN and SHOULD handle it (business errors like InsufficientFunds)
- Unchecked: Programming error or unrecoverable (UserNotFound, InvalidInput)

Exception Chaining (Cause) HOT

// Wrap low-level exception with higher-level context
try {
    // Database call
    userDao.save(user);
} catch (SQLException e) {
    // Don't expose SQL details to caller - wrap it!
    throw new ServiceException("Failed to save user: " + user.getId(), e);
    //                                                                  ^
    //                                                          original cause!
}

// Caller can get the chain:
catch (ServiceException e) {
    System.out.println(e.getMessage());        // High-level message
    System.out.println(e.getCause());          // Original SQLException
    System.out.println(e.getCause().getMessage()); // SQL error details
}
ALWAYS pass the original exception as the cause. Never swallow the root cause - it makes debugging impossible.

finally vs return - Tricky behavior HOT

// Q: What does this return?
int getValue() {
    try {
        return 1;
    } finally {
        return 2; // finally's return OVERRIDES try's return!
    }
}
// Answer: 2

// Q: What does this return?
int getValue() {
    int x = 1;
    try {
        return x; // returns 1 (value captured BEFORE finally runs)
    } finally {
        x = 99;   // modifies x, but return value already captured
    }
}
// Answer: 1 (for primitives, value is copied to return)

// Q: What about objects?
StringBuilder getValue() {
    StringBuilder sb = new StringBuilder("Hello");
    try {
        return sb;
    } finally {
        sb.append(" World"); // Modifies the SAME object!
    }
}
// Answer: "Hello World" (reference was captured, but object is mutable)
NEVER put return in finally block! It swallows exceptions silently and causes confusing behavior.

Suppressed Exceptions (Java 7+)

// When both try AND close() throw exceptions:
try (MyResource r = new MyResource()) {
    r.doWork();     // throws ExceptionA
}
// r.close()        // throws ExceptionB

// ExceptionA is the PRIMARY exception (thrown to caller)
// ExceptionB is SUPPRESSED (attached to ExceptionA)

// Access suppressed exceptions:
catch (Exception e) {
    System.out.println(e);                    // ExceptionA
    for (Throwable suppressed : e.getSuppressed()) {
        System.out.println(suppressed);       // ExceptionB
    }
}

Exception handling in method overriding HOT

class Parent {
    void doWork() throws IOException { }
}

class Child extends Parent {
    // OK: Same exception
    void doWork() throws IOException { }

    // OK: Narrower exception (subclass)
    void doWork() throws FileNotFoundException { }

    // OK: No exception
    void doWork() { }

    // COMPILE ERROR: Broader exception!
    // void doWork() throws Exception { }

    // COMPILE ERROR: New checked exception!
    // void doWork() throws SQLException { }

    // OK: Can add UNCHECKED exceptions
    void doWork() throws NullPointerException { }
}
Rule: Overriding method can throw same, narrower, or no checked exception. Can NOT throw broader or new checked exceptions. Can throw any unchecked exception.

Exception Anti-Patterns (What NOT to do) MUST KNOW

// ANTI-PATTERN 1: Empty catch (swallowing exceptions)
try { riskyCode(); }
catch (Exception e) { } // TERRIBLE! Silent failure. Bug hidden.

// ANTI-PATTERN 2: Catching Exception/Throwable
try { riskyCode(); }
catch (Exception e) { } // Too broad. Catches everything.

// ANTI-PATTERN 3: Using exceptions for flow control
try {
    int i = 0;
    while (true) {
        array[i++]; // Uses ArrayIndexOutOfBounds to exit!
    }
} catch (ArrayIndexOutOfBoundsException e) { }
// Use: for (int i = 0; i < array.length; i++) instead!

// ANTI-PATTERN 4: Logging AND throwing
catch (IOException e) {
    log.error("Error", e);
    throw e; // Exception gets logged TWICE (here + caller)!
}
// Do ONE: log OR throw. Not both.

// ANTI-PATTERN 5: Losing the cause
catch (SQLException e) {
    throw new ServiceException("DB error"); // WRONG: lost cause!
    throw new ServiceException("DB error", e); // RIGHT: keep cause
}

Exception Handling Best Practices MUST KNOW

1. Catch specific exceptions, not generic Exception
2. Use try-with-resources for ALL closeable resources
3. Never swallow exceptions (empty catch block)
4. Always preserve the cause when wrapping exceptions
5. Log OR throw, never both
6. Use custom exceptions for business logic
7. Prefer unchecked exceptions for programming errors
8. Use checked exceptions when caller CAN and SHOULD recover
9. Document exceptions with @throws in Javadoc
10. Fail fast: validate inputs early with Objects.requireNonNull()
11. Don't use exceptions for flow control
12. Don't put return in finally block
13. Catch exceptions at the right level (service layer, not DAO layer)

Exception performance considerations

Creating an exception is EXPENSIVE because it captures the entire stack trace. Don't use exceptions for normal flow control.

Cost of exception:
- Stack trace capture: ~1-5 microseconds
- Normal method call: ~1-10 nanoseconds
- That's 100-1000x slower!

Optimization: If you must throw frequently, override fillInStackTrace() to return this (skip trace capture).

Checked Exceptions in Lambdas / Streams HOT

// Problem: Functional interfaces don't declare checked exceptions
list.stream()
    .map(s -> parseFile(s))  // COMPILE ERROR if parseFile throws IOException
    .collect(toList());

// Solution 1: Wrapper method
private <T, R> Function<T, R> wrap(ThrowingFunction<T, R> fn) {
    return t -> {
        try { return fn.apply(t); }
        catch (Exception e) { throw new RuntimeException(e); }
    };
}
list.stream().map(wrap(this::parseFile)).collect(toList());

// Solution 2: Try-catch inside lambda
list.stream().map(s -> {
    try { return parseFile(s); }
    catch (IOException e) { throw new UncheckedIOException(e); }
}).collect(toList());

// Solution 3: Use libraries (Vavr, jOOL)
// list.stream().map(Unchecked.function(this::parseFile))

Exception Handling in Multithreading HOT

// Thread.run() cannot throw checked exceptions!

// 1. UncaughtExceptionHandler
Thread t = new Thread(() -> { throw new RuntimeException("Oops"); });
t.setUncaughtExceptionHandler((thread, ex) ->
    log.error("Thread {} failed: {}", thread.getName(), ex.getMessage(), ex));
t.start();

// 2. Default handler for ALL threads
Thread.setDefaultUncaughtExceptionHandler((t, e) ->
    log.error("Unhandled: {}", e.getMessage(), e));

// 3. ExecutorService: exceptions captured in Future
Future<?> f = executor.submit(() -> { throw new RuntimeException("fail"); });
try {
    f.get(); // ExecutionException wraps the RuntimeException
} catch (ExecutionException e) {
    Throwable cause = e.getCause(); // the original RuntimeException
}

// 4. CompletableFuture: use exceptionally() or handle()
CompletableFuture.supplyAsync(() -> riskyOp())
    .exceptionally(ex -> fallbackValue);

finally block puzzles INTERVIEW

// Puzzle 1: What's returned?
static int test() {
    try { return 1; }
    finally { return 2; } // ALWAYS returns 2! finally overrides try return
}

// Puzzle 2: When does finally NOT execute?
// 1. System.exit() called
// 2. JVM crash
// 3. Thread killed (Thread.stop() - deprecated)
// 4. Infinite loop in try block

// Puzzle 3: Variable in finally
int x = 0;
try { x = 1; return x; }      // return value (1) is saved BEFORE finally
finally { x = 2; }             // x changes to 2 but return value already saved
// Returns 1! (primitive value was copied)

// But with objects:
StringBuilder sb = new StringBuilder("Hello");
try { return sb; }
finally { sb.append(" World"); }
// Returns "Hello World"! (reference was copied, object is same)

ConcurrentModificationException HOT

// WRONG: modifying collection while iterating
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String s : list) {
    if (s.equals("b")) list.remove(s); // ConcurrentModificationException!
}

// FIX 1: Use Iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove(); // SAFE
}

// FIX 2: removeIf() (Java 8+)
list.removeIf(s -> s.equals("b")); // cleanest way

// FIX 3: Use CopyOnWriteArrayList (concurrent scenarios)
// FIX 4: Collect to remove, then removeAll()

list.removeAll(toRemove);
