Stack Memory in Detail HOT

What is it?
Each thread gets its own private Stack. Every time a method is called, a new Stack Frame is pushed. When the method returns, the frame is popped.
Thread Stack (LIFO): +-------------------+ | methodC() frame | <-- Top (currently executing) | - local var z | +-------------------+ | methodB() frame | | - local var y | +-------------------+ | methodA() frame | | - local var x | +-------------------+ | main() frame | <-- Bottom | - args[] | +-------------------+ Each Stack Frame contains: +------------------------------------+ | Local Variables Array | (primitives + object references) | Operand Stack | (for arithmetic operations) | Frame Data (return address, etc.) | +------------------------------------+ 
void example() {
    int x = 10;                         // x stored on STACK (primitive)
    String name = "John";               // 'name' reference on STACK, "John" in String Pool
    Employee emp = new Employee();     // 'emp' reference on STACK, object on HEAP
}
// When example() finishes: entire stack frame popped, x/name/emp refs are gone
// Employee object on heap is now eligible for GC (no refs pointing to it)
Stack PropertyValue
Default size512KB - 1MB (OS/JVM dependent)
Set with flag-Xss512k or -Xss1m
Thread safetyInherently thread-safe (each thread has own stack)
Error when fullStackOverflowError (usually deep/infinite recursion)

Heap Memory in Detail HOT

What is it?
The Heap is where ALL objects live. It's shared by all threads, and it's managed by the Garbage Collector.
HEAP MEMORY LAYOUT +-----------------------------------------------------------+ | YOUNG GENERATION (~1/3 of heap) | | +-------+ +-------+ +-------+ | | | EDEN | | S0 | | S1 | S0/S1 = Survivor | | | (new | |(from) | | (to) | spaces (one always | | | objs) | | | | | empty) | | +-------+ +-------+ +-------+ | | Objects created here first | | Minor GC happens here (fast, frequent) | +-----------------------------------------------------------+ | OLD GENERATION / TENURED (~2/3 of heap) | | | | Objects that survived many Minor GCs | | Major GC happens here (slow, infrequent) | | | +-----------------------------------------------------------+
Object Lifecycle in Heap:
1. new Employee() --> Born in EDEN 2. Eden fills up --> Minor GC triggers 3. Surviving objects --> Moved to Survivor S0 4. Next Minor GC --> Survivors moved S0 -> S1 (swap each GC) 5. After N survivals (default 15) --> PROMOTED to Old Generation 6. Old Gen fills up --> Major/Full GC triggers (Stop-The-World!)
Heap FlagPurposeExample
-XmsInitial heap size-Xms512m
-XmxMaximum heap size-Xmx2g
-XmnYoung generation size-Xmn256m
-XX:NewRatioRatio of Old/Young-XX:NewRatio=2 (Old is 2x Young)
-XX:SurvivorRatioRatio of Eden/Survivor-XX:SurvivorRatio=8 (Eden is 8x Survivor)
-XX:MaxTenuringThresholdGC cycles before promotion-XX:MaxTenuringThreshold=15

Young Generation (Eden + Survivor Spaces)

What is it?
Where new objects are born. Most objects die young (are short-lived), so GC here is fast and frequent (Minor GC).

Analogy: A nursery in a hospital. Most babies (objects) are born here. Some are healthy (survive) and move to the main ward. Most are short-lived temporary objects.
How Survivor Spaces Work:
Minor GC #1: Minor GC #2: Minor GC #3: Eden: [A][B][C][D] Eden: [E][F][G] Eden: [H][I] S0: (empty) S0: [A(age=1)][C(age=1)] S0: (empty) S1: (empty) S1: (empty) S1: [A(age=2)][E(age=1)] After GC#1: After GC#2: After GC#3: Eden: (cleared) Eden: (cleared) Eden: (cleared) S0: [A][C] (survived) S0: (cleared) S0: [A(age=3)][E(age=2)][H] S1: (empty) S1: [A(age=2)][C(age=2)] S1: (cleared) [E][F] B,D collected (dead) G collected I collected When age reaches threshold (15) --> Object promoted to Old Gen

Old Generation (Tenured)

What is it?
Home for long-lived objects that survived multiple Minor GCs. GC here (Major GC) is slower and causes longer pauses.
Objects that go directly to Old Gen:
ScenarioWhy?
Large objects exceeding threshold-XX:PretenureSizeThreshold
Survivor space overflowToo many survivors, no room in S0/S1
Long-lived objects (age > threshold)Promoted after MaxTenuringThreshold cycles
Dynamic age determinationIf same-age objects exceed 50% of survivor space

Metaspace (replaced PermGen in Java 8) HOT

FeaturePermGen (Java 7 and before)Metaspace (Java 8+)
LocationInside JVM heapNative memory (OS level)
Default sizeFixed (64MB-256MB)Unlimited (grows dynamically)
OOM ErrorPermGen spaceMetaspace
StoresClass metadata, interned strings, static varsClass metadata only
String poolIn PermGenMoved to Heap
Tuning-XX:MaxPermSize-XX:MaxMetaspaceSize
Why was PermGen replaced?
PermGen had a fixed max size, causing frequent OutOfMemoryError: PermGen space in application servers that load/unload classes (like Tomcat). Metaspace uses native memory and grows dynamically, largely eliminating this problem.

What is Garbage Collection? HOT

What is it?
GC is JVM's automatic memory management. It finds and removes objects that are no longer reachable (no live references pointing to them).

Analogy: A cleaning crew in a building. They go through every room (heap) and check which desks (objects) have someone sitting at them (reachable). Empty desks get cleaned up.
Key concepts:
TermMeaning
GC RootsStarting points for reachability: local vars on stack, static fields, active threads, JNI refs
ReachableObject can be reached from a GC root via reference chain
Eligible for GCObject NOT reachable from any GC root
Stop-The-World (STW)All application threads pause while GC runs
Concurrent GCGC runs alongside application threads (less pause)
// When does an object become eligible for GC?
Employee e = new Employee();  // Object created, e points to it
e = null;                      // e no longer points to it -> ELIGIBLE

Employee a = new Employee();
Employee b = new Employee();
a = b;                          // First Employee object -> ELIGIBLE

// Island of isolation
class Node { Node next; }
Node a = new Node();
Node b = new Node();
a.next = b; b.next = a;         // Circular reference
a = null; b = null;             // Both ELIGIBLE (no external refs, GC detects this)

How does GC identify garbage? (Mark and Sweep)

MARK PHASE: Start from GC Roots, traverse all references, MARK reachable objects GC Roots | v [Object A] --ref--> [Object B] --ref--> [Object C] (ALL MARKED as alive) | ref v [Object D] (MARKED) [Object E] (no ref from any root) (NOT marked = GARBAGE) [Object F] --ref--> [Object G] (NOT marked = GARBAGE) SWEEP PHASE: Delete all unmarked objects, free their memory COMPACT PHASE (optional): Move surviving objects together to avoid fragmentation Before: [A][ ][ ][B][ ][C][ ][ ][D] After: [A][B][C][D][ ][ ][ ][ ][ ] (contiguous = faster allocation)

Flag: -XX:+UseSerialGC
FeatureDetail
ThreadsSingle thread for both Minor and Major GC
Best forSmall apps, single CPU, client-side apps
STWYes (all app threads stop)
Heap sizeSmall (< 200MB)
Think of it as one janitor cleaning the whole building alone. Simple but slow for large spaces.

Parallel GC (Throughput Collector)

Flag: -XX:+UseParallelGC (default in Java 8)
FeatureDetail
ThreadsMultiple threads for GC (parallel)
Best forBatch processing, backend jobs where throughput matters
STWYes (but shorter due to parallelism)
GoalMaximize throughput (% of time NOT doing GC)
Think of it as a team of janitors cleaning simultaneously. Faster, but building is still closed during cleaning (STW).

CMS (Concurrent Mark Sweep) - Deprecated

Flag: -XX:+UseConcMarkSweepGC (Deprecated in Java 9, Removed in Java 14)
FeatureDetail
GoalMinimize pause times (low latency)
HowMost GC work done concurrently with app threads
STWOnly during initial-mark and remark (very short)
DownsidesNo compaction (fragmentation), higher CPU usage
Why deprecated?
CMS doesn't compact the heap, leading to fragmentation over time. Eventually triggers a Full GC anyway. G1 GC replaced it as the better low-latency option.

G1 GC (Garbage First) HOTMUST KNOW

Flag: -XX:+UseG1GC (default since Java 9)
G1 divides the heap into REGIONS (not fixed Young/Old): +----+----+----+----+----+----+----+----+ | E | E | S | O | O | H | E |FREE| +----+----+----+----+----+----+----+----+ | O |FREE| E | S | O | O |FREE| E | +----+----+----+----+----+----+----+----+ E = Eden region S = Survivor region O = Old region H = Humongous (large object, > 50% of region) FREE = Empty region Key idea: G1 collects regions with MOST garbage first (hence "Garbage First")
FeatureDetail
GoalBalance between throughput and latency
Pause target-XX:MaxGCPauseMillis=200 (G1 tries to meet this!)
Heap layoutDivided into equal-sized regions (1MB-32MB)
CompactionYes (copies live objects to new regions)
Best forHeap > 4GB, need predictable pause times
Default sinceJava 9

Strong References

// Strong Reference - the default, most common
Object obj = new Object();  // 'obj' is a strong ref
// Object will NEVER be GC'd as long as 'obj' points to it
// Even if JVM runs out of memory -> OutOfMemoryError (won't collect strong refs)

obj = null;  // NOW the object can be collected

Weak References HOT

// Weak Reference - collected at NEXT GC regardless of memory
WeakReference<Employee> weakRef =
    new WeakReference<>(new Employee("John"));

Employee emp = weakRef.get();  // May return null if GC'd!
if (emp != null) {
    // Use emp
}

// Real-world use: WeakHashMap (keys are weak refs)
WeakHashMap<Key, Value> cache = new WeakHashMap<>();
// Entries automatically removed when key has no strong refs

Soft References

// Soft Reference - collected only when memory is LOW (before OOM)
SoftReference<byte[]> softRef =
    new SoftReference<>(new byte[1024 * 1024]);  // 1MB

// Perfect for CACHES: data stays in memory if there's room,
// but gets cleaned up before OOM
byte[] data = softRef.get();
if (data == null) {
    // Cache miss - reload from DB/file
    data = loadFromDB();
}

Phantom References

// Phantom Reference - get() ALWAYS returns null
// Used only to know WHEN an object is about to be collected
// Replacement for finalize() - for cleanup operations
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef =
    new PhantomReference<>(obj, queue);

// When obj is GC'd, its phantom ref appears in the queue
// A cleanup thread can poll the queue and release resources

Can Java have Memory Leaks? HOT

What is it?
YES! Even with GC, Java can have memory leaks. A memory leak in Java means objects that are no longer needed are still REACHABLE (GC can't collect them).

Analogy: Imagine you keep checking out library books but never return them. The library (heap) eventually runs out of books to lend (memory).
24

Common Memory Leak Patterns HOT

// 1. Static collections that keep growing
static List<Object> cache = new ArrayList<>();
void process() {
    cache.add(new BigObject());  // NEVER removed! Grows forever!
}

// 2. Unclosed resources
void readFile() {
    InputStream is = new FileInputStream("file.txt");
    // If exception occurs before close(), resource leaks!
    // FIX: Use try-with-resources
}

// 3. Listeners/Callbacks not removed
button.addActionListener(listener);
// If you never remove it, the button holds a ref to listener forever

// 4. Inner class holding outer class reference
class Outer {
    byte[] bigData = new byte[10_000_000];
    class Inner {
        // Inner implicitly holds ref to Outer (and its 10MB bigData!)
    }
}

// 5. ThreadLocal not cleaned up
static ThreadLocal<UserContext> context = new ThreadLocal<>();
// In thread pools, threads are reused - ThreadLocal values persist!
// FIX: Always call context.remove() in finally block

// 6. String.intern() abuse
for (int i = 0; i < 1_000_000; i++) {
    String s = ("str" + i).intern();  // All interned strings stay in pool!
}

StackOverflowError

// Most common cause: Infinite/deep recursion
void recursive() {
    recursive();  // StackOverflowError! Stack frames keep piling up.
}

// Fix: Add base case, or convert recursion to iteration
void recursive(int depth) {
    if (depth <= 0) return;  // BASE CASE!
    recursive(depth - 1);
}

// Increase stack size if legitimately deep: -Xss2m

Escape Analysis HOT

What is it?
A JIT compiler optimization that determines if an object "escapes" the method. If it doesn't escape, JVM can optimize: 
void calculate() {
    Point p = new Point(3, 4);   // p doesn't escape this method
    double d = p.x * p.x + p.y * p.y;
    // JIT can:
    // 1. Stack Allocation - allocate p on stack instead of heap (no GC!)
    // 2. Scalar Replacement - replace object with its fields: x=3, y=4
    // 3. Lock Elimination - remove synchronization if no escape
}
OptimizationWhat happens
Stack AllocationObject on stack, auto-freed when method returns
Scalar ReplacementObject decomposed into primitive fields on stack
Lock EliminationUnnecessary synchronized blocks removed

String Pool and Memory

HEAP: +---------------------------------------------------+ | String Pool (special area in heap since Java 7) | | +-------+ +-------+ +--------+ | | |"Hello"| |"World"| |"Java" | | | +-------+ +-------+ +--------+ | | | | Regular Heap Objects: | | +------------------+ +------------------+ | | | new String("Hi") | | new String("Hi") | | | +------------------+ +------------------+ | +---------------------------------------------------+ 
String a = "Hello";            // Goes to String Pool
String b = "Hello";            // Reuses SAME object from Pool
String c = new String("Hello"); // NEW object on heap (NOT in pool)
String d = c.intern();         // Returns pool reference (same as a)

a == b     // true  (same pool object)
a == c     // false (different objects)
a == d     // true  (intern() returns pool ref)

finalize() and Cleaner API

finalize() is DEPRECATED since Java 9
Problems: No guarantee when/if it runs, performance overhead, resurrection possible, delays GC. 
// Modern replacement: java.lang.ref.Cleaner (Java 9+)
public class NativeResource implements AutoCloseable {
    private static final Cleaner cleaner = Cleaner.create();
    private final Cleaner.Cleanable cleanable;
    private final long nativePtr;

    public NativeResource() {
        nativePtr = allocateNative();
        cleanable = cleaner.register(this, () -> freeNative(nativePtr));
    }

    @Override
    public void close() { cleanable.clean(); }
}

UNREACHABLE No refs at all --> Eligible for GC 5. COLLECTED GC runs --> Memory freed 6. FINALIZED Cleaner runs --> Post-mortem cleanup (if registered) 7. DEALLOCATED Memory returned --> Available for new objects

Java Memory Model (JMM) and Happens-Before HOT

What is it?
JMM defines how threads interact through memory. Due to CPU caches and compiler optimizations, Thread A's write might not be visible to Thread B immediately. JMM defines rules (happens-before) that guarantee visibility.
Happens-Before RuleMeaning
Program orderEach action in a thread happens-before every subsequent action
Monitor lockUnlock happens-before subsequent lock of same monitor
VolatileWrite to volatile happens-before read of same volatile
Thread startThread.start() happens-before any action in started thread
Thread joinAll actions in thread happen-before join() returns
ConstructorEnd of constructor happens-before finalize()

Memory Best Practices for Production MUST KNOW

Best Practices
1. Set -Xms = -Xmx in production (avoid resizing overhead)
2. Always enable heap dumps on OOM: -XX:+HeapDumpOnOutOfMemoryError
3. Use try-with-resources for all closeable resources
4. Remove ThreadLocal values in finally blocks (thread pool reuse!)
5. Avoid static mutable collections or bound their size
6. Use weak/soft references for caches
7. Monitor GC logs in production: -Xlog:gc*
8. Profile regularly with tools like VisualVM, JProfiler
9. Prefer primitives over wrappers for large arrays (less memory)
10. Use StringBuilder for string concatenation in loops
