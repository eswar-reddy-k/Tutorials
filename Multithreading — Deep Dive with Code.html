Chapter 1

The Burger Shop — One Analogy, Every Concept

Every multithreading term maps to a real kitchen scenario.

🏪 Day 1: 👤 Owner runs the shop alone. One order at a time. Customers wait 20 min. That is single-threaded.

👨‍🍳 Day 2: Hires Cook A, Cook B, Cook C. Three orders handled at once. Wait drops to 7 min. That is multithreading.

🔪 The knife problem: Only one sharp knife. Two cooks grab it simultaneously — knuckle gets cut, burger gets wrong toppings. That is a race condition / data corruption.

🗒️ The sticky-note fix: Post a note on the knife: "In use — Cook A." Others wait. That is a mutex (lock).

🚫 The Tuesday freeze: Cook A holds the knife, waiting for the cutting board. Cook B holds the cutting board, waiting for the knife. Nobody moves for 30 minutes. That is a deadlock.

📦 The ingredient shelf: 🚚 Supplier restocks a 5-box shelf. Cooks grab from it. Shelf full → supplier waits. Shelf empty → cooks wait. That is the Producer-Consumer pattern.

⚡ Why it matters

Modern hardware has 4–128 CPU cores. A single-threaded app uses ONE. Multithreading lets you use all of them.

16 cores × 16 independent tasks = up to 16× throughput — when coordination cost is low.
16 cores with every task waiting on ONE lock = effectively single-threaded, sometimes slower due to overhead.
Chapter 2

Prerequisites with Code

Five building blocks. Each with an analogy and a code example.

Process vs Thread

Process = entire running app with its own private memory. Thread = lightweight worker inside a process, sharing its heap with other threads.

// One process, two threads sharing heap object "counter" Counter counter = new Counter(); // lives on HEAP — shared Thread t1 = new Thread(() -> counter.increment()); Thread t2 = new Thread(() -> counter.increment()); t1.start(); // both threads operate on the same counter object t2.start(); t1.join(); t2.join(); System.out.println(counter.value); // might NOT be 2 — race condition!
# Python: threads share heap, each has own stack import threading shared_list = [] # heap — shared between all threads def worker(n): local_var = n * 2 # stack — private to this thread, always safe shared_list.append(local_var) # heap write — needs protection threads = [threading.Thread(target=worker, args=(i,)) for i in range(5)] for t in threads: t.start() for t in threads: t.join()
Analogy: Process = an apartment building with private units. Threads = tenants sharing the lobby and kitchen (heap) but having private rooms (stack).

Stack vs Heap — The Root of Most Race Conditions

📦 Stack — thread-local, always safe

Local variables inside a method. No other thread can see them. Zero lock needed.

void compute() { int result = 0; // stack — only this thread sees it result += 42; // perfectly safe, no lock needed return result; }

🌐 Heap — shared, needs protection

Object fields, class fields, globals. Multiple threads read/write simultaneously. This is where bugs live.

class Counter { int count = 0; // heap — ALL threads share this } // Thread 1: count++ ← these three lines // Thread 2: count++ ← run simultaneously // Result: possibly 1 instead of 2 ← RACE!
Rule: local variable = safe. Field on an object = protect it.

CPU Scheduler

The OS decides which thread runs on which core and for how long (a time slice, ~1–10 ms). You cannot predict or control this order.

// The scheduler can pause a thread ANYWHERE — even mid-statement // This is why count++ can cause bugs: scheduler pauses after READ, before WRITE count++; // Compiles to: // 1. LOAD reg = count ← scheduler can pause HERE // 2. ADD reg = reg + 1 ← or HERE // 3. STORE count = reg ← or before this completes
CPU-bound rule: pool size ≈ number of cores. IO-bound rule: pool size ≈ cores / (1 − wait_ratio). With 90% IO wait: 8 cores → ~80 threads optimal.

Context Switch

Switching from Thread A to Thread B requires saving A's full CPU state and loading B's. Fast, but not free.

// Approximate costs (nanoseconds) Context switch cost: ~1,000 – 10,000 ns // 1–10 µs L1 cache miss: ~4 ns L3 cache miss: ~40 ns RAM access: ~100 ns // If each task = 50µs and a switch = 5µs → 10% overhead // If each task = 1µs and a switch = 5µs → switch costs MORE than the work!
Too many tiny tasks = more time switching than working. Batch small work into larger chunks before parallelising.

Concurrency vs Parallelism

🔀 Concurrency — design

Tasks can make progress — even on 1 core, by interleaving. About structure.

# Single-core phone: music + download "at the same time" # OS switches every few ms — feels simultaneous Thread(play_music).start() Thread(download_file).start()

⚡ Parallelism — hardware

Tasks run at literally the same instant on different cores. Needs multiple cores.

Chapter 4

Race Condition — Exact CPU-Level Trace

Not hand-wavy — every instruction that causes the bug, shown step by step.

Problem: Two threads each call count++ 1000 times. Expected: 2000. Actual: random value like 1743 or 1891.

Why? count++ is NOT one instruction. It is three CPU operations that can be interrupted between any two.

// count++ compiles to THREE operations: // 1. READ: temp = count (load from RAM into CPU register) // 2. ADD: temp = temp + 1 (CPU arithmetic) // 3. WRITE: count = temp (store result back to RAM) // // The OS scheduler can PAUSE a thread between ANY two of these steps.

🔬 Step-by-step interleaving — count starts at 5, expected result: 7

TimeThread AThread Bcount
T=1READ temp_A = 5idle5
T=2ADD temp_A = 6idle5
T=3⏸ Scheduler PAUSES Thread AScheduler STARTS Thread B5
T=4pausedREAD temp_B = 5 (stale!)5
T=5pausedADD temp_B = 65
T=6pausedWRITE count = 66
T=7WRITE count = 6 (stale temp_A!)done6 ❌
Resultcount = 6 — one increment lost! Expected 7.
The bug is invisible to both threads. Each believed it incremented correctly. The corruption comes from interleaving, not from any single thread's logic.

✅ Three ways to fix it

// Fix 1: synchronized — only ONE thread in the critical section at a time class SafeCounter { private int count = 0; synchronized void increment() { // ONLY ONE THREAD AT A TIME count++; // now the 3 steps are serialised } synchronized int get() { return count; } } // OR use ReentrantLock for more control private final Lock lock = new ReentrantLock(); void increment() { lock.lock(); try { count++; } finally { lock.unlock(); } // ALWAYS in finally — or lock is never released! }
// Fix 2: AtomicInteger — hardware-level CAS (Compare-And-Swap) // Single uninterruptible CPU instruction — no lock needed import java.util.concurrent.atomic.*; class AtomicCounter { private final AtomicInteger count = new AtomicInteger(0); void increment() { count.incrementAndGet(); } int get() { return count.get(); } // CAS manually (optimistic locking pattern) void safeMultiply(int factor) { int prev, next; do { prev = count.get(); next = prev * factor; } while (!count.compareAndSet(prev, next)); // retry if changed since read } } // AtomicLong, AtomicBoolean, AtomicReference all exist too
// Fix 3: eliminate shared mutable state — each thread works on private data // then combine at the end in ONE thread int[] partials = new int[threadCount]; // Each thread writes ONLY to its own slot — no sharing, no lock needed for (int i = 0; i < threadCount; i++) { final int idx = i; pool.submit(() -> { for (int j = 0; j < 1000; j++) partials[idx]++; // safe! }); } pool.shutdown(); pool.awaitTermination(1, TimeUnit.MINUTES); // Merge in one place (sequential, safe) int total = Arrays.stream(partials).sum(); // always 2000
# Python: use threading.Lock() import threading count = 0 lock = threading.Lock() def safe_increment(): global count with lock: # auto-acquires and auto-releases even on exception count += 1 threads = [threading.Thread(target=safe_increment) for _ in range(1000)] for t in threads: t.start() for t in threads: t.join() print(count) # always 1000 # Or use concurrent.futures for map-reduce pattern (no lock needed) from concurrent.futures import ThreadPoolExecutor with ThreadPoolExecutor(4) as ex: parts = list(ex.map(lambda _: 1, range(1000))) total = sum(parts) # no shared mutable state at all

Deadlock — Step-by-Step with Code

The most catastrophic bug. Apps freeze silently with no error or exception.

Scenario: A banking app transfers money. Transfer A→B: lock account A then B. Transfer B→A: lock account B then A. Start both simultaneously.

TimeThread 1: Transfer A→BThread 2: Transfer B→A
T=1🔒 Locks Account Aidle
T=2holds A🔒 Locks Account B
T=3⏳ Tries to lock B — BLOCKED (Thread 2 holds it)holds B
T=4waiting…⏳ Tries to lock A — BLOCKED (Thread 1 holds it)
∞💀 Both wait forever. App frozen. No exception thrown.
// ❌ DEADLOCK — different lock acquisition order void transferAtoB(Account a, Account b, int amount) { synchronized(a) { // Thread 1 locks A first synchronized(b) { // Thread 1 waits for B ← BLOCKS if Thread 2 holds B a.debit(amount); b.credit(amount); } } } void transferBtoA(Account b, Account a, int amount) { synchronized(b) { // Thread 2 locks B first synchronized(a) { // Thread 2 waits for A ← BLOCKS if Thread 1 holds A b.debit(amount); a.credit(amount); } } } // Run both simultaneously → DEADLOCK
// ✅ FIX: Always acquire locks in the same global order (by account ID) // This breaks the circular wait — the 4th Coffman condition void transfer(Account from, Account to, int amount) { Account first = from.id < to.id ? from : to; // lower ID always first Account second = from.id < to.id ? to : from; synchronized(first) { synchronized(second) { from.debit(amount); to.credit(amount); } } } // Both Transfer A→B and Transfer B→A now lock in ID order. // No thread can ever wait for a lock held by a lower-priority thread.
// ✅ FIX: tryLock with timeout — if can't acquire both, back off and retry boolean transfer(Account from, Account to, int amount) throws InterruptedException { while (true) { if (from.lock.tryLock(50, MILLISECONDS)) { try { if (to.lock.tryLock(50, MILLISECONDS)) { try { from.debit(amount); to.credit(amount); return true; // success } finally { to.lock.unlock(); } } } finally { from.lock.unlock(); } } Thread.sleep(10 + (long)(Math.random() * 20)); // jitter before retry } }
# Python: use consistent lock ordering to prevent deadlock import threading def transfer(from_acc, to_acc, amount): # Always lock the lower-id account first first, second = sorted([from_acc, to_acc], key=lambda a: a.id) with first.lock: # acquire in consistent order with second.lock: from_acc.balance -= amount to_acc.balance += amount # Both directions now use the same order → no circular wait → no deadlock
All four must hold simultaneously. Break any one to prevent deadlock.

1. Mutual exclusion — resource held by only one thread at a time.
2. Hold and wait — thread holds a resource while waiting for another.
3. No preemption — resources can't be forcibly taken away.
4. Circular wait — a cycle in the wait-for graph.

Lock ordering breaks #4. tryLock timeout breaks #2 (back off = release what you hold).

Synchronization Primitives — With Full Code

Six tools, six problems. Pick the right one.

// Mutex: ONE thread at a time in the critical section // Use when: multiple threads write to shared state private final ReentrantLock lock = new ReentrantLock(); private int balance = 0; void deposit(int amount) { lock.lock(); try { balance += amount; } finally { lock.unlock(); } // must unlock even if exception thrown } int getBalance() { lock.lock(); try { return balance; } finally { lock.unlock(); } } // Python equivalent # lock = threading.Lock() # with lock: # balance += amount
// Read-Write Lock: many readers OR one writer (never both) // Use when: reads >> writes (e.g. config, cache, lookup tables) private final ReadWriteLock rwl = new ReentrantReadWriteLock(); private Map<String, String> config = new HashMap<>(); // Multiple threads can read simultaneously String get(String key) { rwl.readLock().lock(); try { return config.get(key); } finally { rwl.readLock().unlock(); } } // Only ONE thread can write (blocks all readers while writing) void set(String key, String val) { rwl.writeLock().lock(); try { config.put(key, val); } finally { rwl.writeLock().unlock(); } } // Python: threading.RLock() for reentrant lock # Or use readers-writer lock from third-party libs
// Semaphore: allow at most N threads at once // Use when: limiting access to a bounded pool of resources // Limit database connections to 10 private final Semaphore dbPool = new Semaphore(10); // 10 permits void queryDB(String sql) throws InterruptedException { dbPool.acquire(); // blocks if 10 connections already open try { db.execute(sql); } finally { dbPool.release(); } // always free the permit } // Python # import threading # sem = threading.Semaphore(10) # with sem: # db.execute(sql) // Semaphore(1) = binary semaphore = equivalent to a mutex // Semaphore(0) = signalling: producer releases, consumer acquires
// Condition Variable: wait for a condition to become true // Classic use: bounded producer-consumer buffer final Object lock = new Object(); Queue<Item> buf = new LinkedList<>(); final int MAX = 5; // Producer void produce(Item item) throws InterruptedException { synchronized(lock) { while (buf.size() == MAX) // ← WHILE not IF (spurious wakeups!) lock.wait(); // releases lock and sleeps buf.add(item); lock.notifyAll(); // wake sleeping consumers } } // Consumer Item consume() throws InterruptedException { synchronized(lock) { while (buf.isEmpty()) // ← WHILE not IF lock.wait(); Item item = buf.poll(); lock.notifyAll(); // wake sleeping producers return item; } }
// Atomic operations: hardware CAS — lock-free for single variables // Use when: counters, flags, single-reference swaps AtomicInteger count = new AtomicInteger(0); AtomicBoolean running = new AtomicBoolean(true); AtomicLong total = new AtomicLong(0L); count.incrementAndGet(); // thread-safe ++ count.addAndGet(5); // thread-safe +=5 running.compareAndSet(true, false); // atomic if-equal-then-set total.getAndAdd(100L); // returns old value, adds 100 // CAS loop for compound updates long multiplyBy(int factor) { long old, newVal; do { old = total.get(); newVal = old * factor; } while (!total.compareAndSet(old, newVal)); // retry if changed }
// CountDownLatch: wait for N events to complete CountDownLatch latch = new CountDownLatch(3); for (int i = 0; i < 3; i++) { pool.submit(() -> { doWork(); latch.countDown(); // decrement counter when done }); } latch.await(); // main thread blocks until count reaches 0 System.out.println("All 3 tasks finished!"); // CyclicBarrier: all threads meet at a checkpoint before continuing CyclicBarrier barrier = new CyclicBarrier(4, () -> System.out.println("Phase done")); // In each worker thread: doPhaseWork(); barrier.await(); // waits until all 4 threads reach this line doNextPhaseWork();

// Fork-Join: divide data in half recursively, process each half in parallel // Java ForkJoinPool is used internally by parallel streams class SumTask extends RecursiveTask<Long> { private static final int THRESHOLD = 1000; private final int[] arr; private final int lo, hi; SumTask(int[] arr, int lo, int hi) { this.arr=arr; this.lo=lo; this.hi=hi; } @Override protected Long compute() { if (hi - lo < THRESHOLD) // base case: small enough → do sequentially return sumSequential(lo, hi); int mid = lo + (hi - lo) / 2; SumTask left = new SumTask(arr, lo, mid); SumTask right = new SumTask(arr, mid, hi); left.fork(); // start left in thread pool return right.compute() + left.join(); // run right here, wait for left } } ForkJoinPool fjp = ForkJoinPool.commonPool(); long total = fjp.invoke(new SumTask(arr, 0, arr.length)); // Shorthand: Java 8+ parallel streams use ForkJoinPool automatically long total2 = IntStream.of(arr).parallel().sum();

Memory Model — Visibility and Ordering

The hardest concept. Modern CPUs cache data per-core and reorder instructions. This breaks multithreaded code in unexpected, non-deterministic ways.

The invisible bug: Thread A sets ready = true after writing data = 42. Thread B loops until ready == true then reads data. Looks correct. But Thread B can spin forever — or read data = 0 even after ready is true. No lock was violated. No exception thrown.

Why? The CPU cached ready in Thread B's L1 cache. Or the compiler reordered Thread A's writes for performance. Thread B never sees the update.

// ❌ BUG: no visibility guarantee for ready or data boolean ready = false; int data = 0; // Thread A (writer) data = 42; // write 1 — may be reordered AFTER ready=true by CPU! ready = true; // write 2 // Thread B (reader) // Problem 1: ready may be cached as false in B's L1 cache → infinite loop // Problem 2: even if B sees ready=true, it may still read data=0 // because write 1 was reordered to happen after write 2 while (!ready) {} // may never exit System.out.println(data); // may print 0 — data write not yet visible
// ✅ FIX: volatile keyword // 1. Forces reads/writes to go to main memory (bypasses CPU cache) // 2. Prevents instruction reordering around the volatile variable volatile boolean ready = false; volatile int data = 0; // Thread A data = 42; // guaranteed to be written to main memory BEFORE ready ready = true; // volatile write creates memory barrier // Thread B while (!ready) {} // always reads from main memory — will eventually see true System.out.println(data); // guaranteed to see 42 (happens-before) // volatile is NOT enough for compound operations (check-then-act) // volatile int x = 0; then x++ is still NOT thread-safe! // volatile only protects single reads/writes // Use AtomicInteger or synchronized for compound updates
// Java Memory Model: Happens-Before guarantees // If X happens-before Y, all effects of X are visible when Y executes // Rule 1: Thread start // Everything before t.start() is visible to t when it begins data = 42; // ← visible to t t.start(); // Rule 2: Thread join // Everything t does is visible after t.join() returns t.join(); System.out.println(sharedVar); // safe to read t's writes // Rule 3: Lock release → acquire // Everything before unlock() is visible after the next lock() on same lock lock.lock(); data = 99; lock.unlock(); // Thread A lock.lock(); System.out.println(data); lock.unlock(); // Thread B sees 99 // Rule 4: volatile write → read // Everything before volatile write is visible after volatile read of same var // Rule 5: Static initializers // Class static fields are safely visible to all threads after class load class Cfg { static final int MAX = 100; } // always visible, no lock needed
// Classic trap: Double-Checked Locking (DCL) without volatile // ❌ BROKEN: partial construction visible across threads class Singleton { private static Singleton instance; // ← missing volatile! static Singleton get() { if (instance == null) { // check without lock synchronized(Singleton.class) { if (instance == null) // double-check inside lock instance = new Singleton(); // BUG: partial init visible! } } return instance; } } // ✅ FIX 1: add volatile private volatile static Singleton instance; // volatile fixes it // ✅ FIX 2 (best): Initialization-on-demand holder — no locks, no volatile class Singleton { private static class Holder { static final Singleton INSTANCE = new Singleton(); } static Singleton get() { return Holder.INSTANCE; } } // JVM class loading guarantees thread safety of static initializers

When and Where to Use Multithreading

Use it intentionally, not reflexively.

WorkloadStrategyWhy
UI + long background task1 background thread or asyncKeep UI responsive without complex sharing
Web server (many requests)Thread pool + non-blocking IOEach request gets a thread; IO releases it fast
CPU-bound batch (image resize)ForkJoinPool / parallel streamUse all cores; tasks are independent
Event streaming (Kafka, queues)Producer-consumer + BlockingQueueDecouples produce/consume speed mismatch
Database per-requestConnection pool (Semaphore)Bounded resource — limit concurrent DB connections
Simple single-user scriptSingle threadComplexity overhead > gains
Read-heavy cache / configvolatile or ReadWriteLockRead-heavy: avoid exclusive lock on every read

🚦 Quick decision checklist

  • ☑ Tasks are independent → good parallelism candidate
  • ☑ Shared state is minimal → lower race risk
  • ☑ IO wait ratio is high → use many more threads than cores
  • ☑ Work chunk is large enough → context-switch overhead stays small
  • ☑ You can test it under concurrent load → thread bugs are timing-dependent
